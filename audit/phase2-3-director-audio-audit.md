# Phase 2/3 — Director/Camera + Audio-Analysis Pipeline: Risk Audit

## 1. Director / Camera Logic

**How it actually works:** Camera moves are decided in `lofi-diorama.ts` by `updateCameraSequencer()` (`:3474-3615`), which each frame picks the active `MacroShot` for the current playback time, optionally overrides with an `MicroCut`, and either snaps or lerps the camera toward a computed position/target. There are two independent camera-resolution code paths — `snapToCamera()` (`:3440-3472`, used by the editor's "double-click to preview" and manual snap) and the inline logic inside `updateCameraSequencer()` (`:3507-3559`, used at actual playback/render time) — that duplicate the same mood→offset switch statement.

Ranked findings:

### High risk — Camera targeting resolves object names by fuzzy string matching, with a silent no-op fallback
`findGearObject()` (`lofi-diorama.ts:3390-3434`) strips a target ID down to `[a-z0-9]`, tries exact match, then falls back to **substring partial-matching** ("if either cleaned string contains the other, length ≥ 3"), then a deep-tree exact-match pass. This means a mistyped or ambiguous target ID (e.g. two device names sharing a substring) can silently resolve to the wrong 3D object with no warning surfaced to the user — only a `console.warn` (`:3525-3528`) that nobody sees outside devtools. If `findGearObject` returns `null` (target empty, or genuinely unmatched), the camera falls back to a hardcoded desk-center point `(0, 5.6, -7)` — the storyboard shot is silently degraded to a generic wide shot instead of failing loudly. Since target IDs are chosen by the audio analyzer worker at random from `primaryArray`/`secondaryArray` (see §2), a single bad ID in the device-selection UI can quietly turn many generated shots into "point at desk" shots.

### Medium-high — Mood→camera-offset logic is duplicated in two places and will drift
The `switch (mood) { case 'balanced' / 'submerged' / 'chaotic' / 'ambient' }` offset table exists **twice**, near-identically, in `snapToCamera()` (`:3455-3461`) and `updateCameraSequencer()` (`:3535-3553`). There's no shared helper. Any future tuning of the mood camera language (e.g. adjusting the `ambient` B-roll framing) has to be made in both places or the editor preview (`snapToCamera`) and the actual render (`updateCameraSequencer`) will silently diverge — what you see when double-clicking a storyboard panel in the editor won't match what renders.

### Medium — Transition timing is hardcoded, not audio- or duration-aware
`transitionDuration` is fixed to `400ms` (whip-pan) or `1500ms` (smooth) (`:3572`), regardless of the shot's actual `duration` or the track's tempo. A macro shot shorter than 1.5s (the analyzer's `minMacroDuration` is 1.0s, see §2) will spend most or all of its screen time still transitioning, never settling into the "solid lock" state — a storyboard-quality bug that's easy to trigger from analyzer output without any validation catching it.

### Medium — No randomness guard for consecutive same-target macro shots
When the diorama-analyzer worker assigns a macro shot's target, it does `arrayToUse[Math.floor(Math.random() * arrayToUse.length)]` (`diorama-analyzer.worker.ts:163-165`) with no "don't repeat the last pick" logic. Two or more consecutive macro shots can land on the identical target/camera framing back-to-back — the director has no cut-variety enforcement, so a generated storyboard can visually stall on one device for an extended span purely by bad luck, especially when `primaryArray`/`secondaryArray` contain few devices.

### Where to add manual creative control (informational, not a defect)
The cleanest existing seam is already there: `AudioDirector` dispatches `capture-camera` / `snap-camera` events (`AudioDirector.ts:996-999, 909`), handled in `lofi-dashboard.ts:1199-1247`, which write a `cameraPos`/`cameraLookAt` pair onto a `MacroShot`. When present, both camera-resolution paths use it directly and skip the mood/target-lookup heuristics entirely (`lofi-diorama.ts:3445-3447, 3511-3513`). This is a per-shot manual override already wired end-to-end (editor → export → render). What's missing for finer control: no override for `transitionDuration`/easing beyond the 3-value enum, no way to keyframe within a shot (camera is locked/static for the whole macro-shot duration once resolved), and no per-microcut manual camera override (micro cuts always use the hardcoded top-down `(0,14,0.1)` offset, `:3586`, with no capture mechanism analogous to macro shots).

## 2. Audio-Analysis → Storyboard Pipeline

**What's actually extracted:** Per 512-sample frame, `diorama-analyzer.worker.ts` computes three Meyda features — `rms` (energy), `zcr` (zero-crossing rate, a percussiveness/noisiness proxy), and `spectralCentroid` (brightness). There is **no tempo/BPM detection, no true "beat" tracking, and no genre/mood classifier** — "mood" is a post-hoc bucket derived from just RMS and spectral centroid relative to track-wide averages. This is worth flagging on its own: what's presented conceptually as mood/beat/energy analysis is, mechanically, a transient/energy-delta detector with a 4-bucket brightness/loudness heuristic layered on top — call the reliability ceiling accordingly.

Ranked findings:

### High risk — Global (non-windowed) adaptive thresholding will misbehave on dynamic tracks
The transient threshold (`transientMultiplier` scaled by `sensitivity`, `:112-117`) and the ZCR threshold are both computed **once, globally**, from the median/stddev of the entire track's energy-delta and ZCR distributions. For a track with a quiet intro/verse and a loud chorus (extremely common), the global threshold will be dominated by the loud sections: quiet-section transients may never cross it (few/no micro cuts or macro-shot breaks during quiet parts), while the loud section may fire excessively. There is no per-region/windowed re-normalization. This is the single most likely cause of a storyboard that looks great on one track and badly unbalanced (all cuts clustered in the chorus, dead editorial silence in the verse) on another, with no signal to the user that this happened.

### High risk — Mood thresholds are unnormalized magic numbers, not perceptually calibrated
Mood classification (`:179-186`, duplicated `:232-239`) uses fixed multipliers: `avgRms < maxRMS * 0.15` → ambient; `avgCentroid > globalAverageCentroid * 1.3` → chaotic; `avgCentroid < globalAverageCentroid * 0.7` → submerged; else balanced. These thresholds were presumably tuned against one or a few reference tracks and have no documented basis — they're relative to the *current track's own* max/average, which helps somewhat, but tracks with narrow dynamic range (e.g. heavily compressed/mastered EDM) may never produce `ambient` (never dip below 15% of their own max), and tracks with a naturally narrow spectral range may never produce `chaotic`/`submerged`. Worth flagging: mood diversity is not guaranteed and can silently collapse to `balanced` for entire tracks.

### Medium-high — Intensity calculation uses an absolute (non-normalized) scale, inconsistent with the mood logic next to it
`intensity = clamp(avgRms * 3.0, 0, 1)` (`:188, 241`) is a flat multiplier against absolute RMS, whereas the mood calculation two lines above is relative to the track's own `maxRMS`. This inconsistency means a quiet, dynamically-compressed track could have every macro shot's `intensity` pinned near 0 (or near 1, if mastered hot) regardless of its actual relative loudness within the track — `intensity` isn't guaranteed to correlate meaningfully with the mood bucket it's paired with in the same shot object.

### Medium — `minMacroDuration` (1.0s) is a fixed constant, independent of `sensitivity`
Higher `sensitivity` lowers the transient threshold, producing more/shorter steady-state gaps between transients — but `minMacroDuration = 1.0` (`:141`) doesn't scale down with it. At high sensitivity on a busy track, most steady-state blocks between frequent transients will fall under 1.0s and get silently dropped entirely (`:156, 210` — the `if (duration >= minMacroDuration)` guard just discards them, no fallback). The practical failure mode: cranking sensitivity up can produce a storyboard with many micro cuts but zero or very few macro shots, i.e. never establishing any base camera position for the sequencer to control — worth checking against how `updateCameraSequencer` behaves when `macroShots` is empty (it returns early at `:3475` and does nothing, leaving the camera wherever it last was, which for `isRenderMode` is likely the default orbit position).

### Medium — Zero-frame / short-audio edge case is only partially guarded
`if (totalFrames < 2) { return empty macroShots/microCuts }` (`:55-62`) protects against very short/empty audio, but nothing checks for a fully-silent buffer that passes the `totalFrames >= 2` check (e.g. a mostly-silent track with `rms` near zero throughout, `maxRMS` near zero). `avgRms < maxRMS * 0.15` degenerates when `maxRMS` itself is ~0 — this doesn't crash, but the mood/intensity values become numerically meaningless (dividing near-zero by near-zero-scaled thresholds), and the transient/ZCR thresholds derived from near-silent noise floors are effectively random. No detection or warning for "this track has almost no signal" exists before analysis runs.

### Medium — Random target assignment has no distribution guarantee, feeding directly into the camera-repeat risk noted in §1
`primaryArray`/`secondaryArray` are alternated per macro shot (odd/even counter, `:158-161`), but the actual device *within* whichever array is picked via unweighted `Math.random()`. Combined with §1's lack of anti-repeat logic, a short `primaryArray` (e.g. 2 devices) makes back-to-back-to-back identical shot targets a real (not edge-case) probability, not just a rare occurrence.

### Low-medium — Sensitivity's effect on `microHoldOff` (0.3s → 0.1s) is small relative to typical beat spacing
At max sensitivity, the minimum gap between micro cuts is still 0.1s (10 cuts/sec ceiling) — reasonable as a hard safety floor, but there's no coupling to actual detected tempo, so at high sensitivity on fast/busy material the cuts can still arrive faster than is visually parseable as "cuts" (approaching a strobe rather than an edit), with nothing in the pipeline aware of a target cuts-per-second budget.
